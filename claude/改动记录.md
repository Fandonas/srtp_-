 问题一、 修改内容（semantic_alignment_few_shot.py 第 109~113 行）

  在 _ensure_text_features_initialized() 中，_create_stage_text_features() 执行完毕后新增了 4 行：

  del self._full_clip_model
  self._full_clip_model = None
  if torch.cuda.is_available():
      torch.cuda.empty_cache()

  为什么这样做是安全的：

  1. _full_clip_model 的唯一作用是调用 encode_text() 提取各类别各阶段的文本特征向量
  2. 提取结果已全部缓存在 self._stage_text_features（字典，只存 Tensor，不含模型）
  3. 后续训练/推理中所有文本特征查询都走 self._stage_text_features[class_name]，不再需要模型本身
  4. self._full_clip_model = None 确保引用被清除，Python GC 可以回收该对象
  5. torch.cuda.empty_cache() 将 PyTorch 缓存的显存归还给 CUDA，使 nvidia-smi 立即可见显存下降

  预期效果：ViT-B/16 对应的完整 CLIP 模型（视觉编码器 ~344MB + 文本编码器 ~252MB）在初始化后即被释放，后续训练全程节省约 596MB 显存。

  问题4修复说明

  核心问题回顾

  评估时 context2(target) 被调用两次：
  1. 第1次：super().forward() 内部（父类 few_shot.py 第 2974 行）
  2. 第2次（冗余）：子类 forward() 中显式调用（原第 632-633 行）

  两次调用完全相同：context2(target_raw, target_raw, target_raw)，都是 Temporal Transformer（dim=512, 8头）的前向计算，对 [10, 4, 512] 张量做 self-attention，额外占用激活显存和计算时间。

  修复策略

  新增路径判断（检测是否为非默认评估模式）：

  use_eval_text = hasattr(self.args.TRAIN, "EVAL_TEXT") and self.args.TRAIN.EVAL_TEXT
  use_combine   = hasattr(self.args.TRAIN, "COMBINE")    and self.args.TRAIN.COMBINE

  两条分支：

  ┌──────────────────────────────┬──────────────────┬────────────────────────────────────────────────┐
  │             路径             │       条件       │                    处理方式                    │
  ├──────────────────────────────┼──────────────────┼────────────────────────────────────────────────┤
  │ 非默认（EVAL_TEXT/COMBINE）  │ 使用上述任一标志 │ 保留 super().forward() + 单独 context2(target) │
  ├──────────────────────────────┼──────────────────┼────────────────────────────────────────────────┤
  │ 默认 OTAM 路径（我们的配置） │ 无上述标志       │ 内联父类 OTAM 逻辑，context2(target) 只算1次   │
  └──────────────────────────────┴──────────────────┴────────────────────────────────────────────────┘

  - get_feats（ViT-B/16 backbone 前向）：评估时 2次 → 1次（Problem 2已从3→2，此处再从2→1）
