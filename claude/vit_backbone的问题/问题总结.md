---
  二、根本原因逐层分析

  原因 1：ViT 的 CLIP 预训练特征本身已近饱和（零样本性能即是上限）

  在代码中 CNN_OTAM_CLIPFSAR.__init__ 里：

  # few_shot.py:2710-2718
  elif cfg.VIDEO.HEAD.BACKBONE_NAME=="ViT-B/16":
      backbone, self.preprocess = load(cfg.VIDEO.HEAD.BACKBONE_NAME, ...)
      self.backbone = backbone.visual
      self.mid_dim = 512

  CLIP ViT-B/16 是比 CLIP RN50 能力强得多的视觉编码器（零样本 ImageNet 精度 ~68% vs ~59%）。它输出的 512 维视觉特征已经与文本特征高度对齐，天然具备跨类别的泛化能力。

  举具体例子：5-way 1-shot 验证集任务中，CLIP ViT-B/16 在"没有任何训练"时就已能达到 ~74% 准确率（25.8% 错误率）。而 RN50 出发点是 ~60% 准确率（40% 错误率）。

---
  原因 2：微调使 ViT 特征灾难性遗忘（Catastrophic Forgetting）

  训练循环（train_net_few_shot.py:104-134）在 USE_CLASSIFICATION=True 时执行：

  loss = (F.cross_entropy(model_dict["logits"], task_dict["target_labels"].long()) +
          cfg.TRAIN.USE_CLASSIFICATION_VALUE * F.cross_entropy(
              model_dict["class_logits"],
              torch.cat([real_support_labels, real_target_labels]).long()))
          / cfg.TRAIN.BATCH_SIZE

  这个损失通过梯度更新以下参数：
  - self.backbone（ViT 视觉编码器，全量参数）
  - self.context2（Transformer_v1，dim=512）
  - self.query_embed_train（Embedding(31, 512)）
  - self.scale（标量）

  问题的核心：用为 RN50 设计的同一个 BASE_LR: 0.00001 来微调 ViT，LR 对 ViT 来说过大。

  - ViT 的全局自注意力机制：每一个参数更新都会影响到所有位置的特征，特征空间高度耦合。轻微的梯度就可以打乱整个 CLIP 对齐结构。
  - RN50 的卷积特征：局部感受野，每个卷积核只影响局部区域，特征更鲁棒，可以从"差的初始点"通过训练稳定改进。

  这就是为什么 epoch 2 时 ViT 从 25.8% 跳到 30.5%（特征质量下降），而 RN50 的 epoch 2 从 35.5% 降到 30.3%（特征质量提升）。

---
  原因 3：mid_dim 设定正确但掩盖了维度对齐问题

  # few_shot.py:2704-2718
  if cfg.VIDEO.HEAD.BACKBONE_NAME=="RN50":
      self.mid_dim = 1024    # RN50 输出 1024 维
  elif cfg.VIDEO.HEAD.BACKBONE_NAME=="ViT-B/16":
      self.mid_dim = 512     # ViT 输出 512 维

  context2 = Transformer_v1(dim=self.mid_dim, heads=8, dim_head_k=self.mid_dim//8, ...)

  但注意 Transformer_v1 的默认参数：

  # few_shot.py:983
  class Transformer_v1(nn.Module):
      def __init__(self, heads=8, dim=2048, dim_head_k=256, dim_head_v=256,
                   dropout_atte=0.05, mlp_dim=2048, ...):  # mlp_dim 默认 2048！

  调用时：Transformer_v1(dim=512, heads=8, dim_head_k=64, dropout_atte=0.2) → mlp_dim 使用默认值 2048！

  这意味着 ViT 的 context2 内部 FeedForward 网络是 Linear(512→2048)→GELU→Linear(2048→512)，内部维度是输入的 4 倍，参数量极大，容量远超 RN50 的 FeedForward(1024→2048→1024)（倍数仅 2
  倍）。这会让 context2 在 ViT 上有更大的过拟合风险，但不是主要原因。

---
  原因 4：SEMANTIC_COMBINE: true 在 ViT 上的干扰

  评估时（_fuse_semantic_and_visual_probs_eval）执行：

  # semantic_alignment_few_shot.py:497
  fused_dists = -(semantic_probs.pow(self.args.TRAIN.TEXT_COFF) *
                  visual_probs.pow(1.0 - self.args.TRAIN.TEXT_COFF))
  model_dict["logits"] = -fused_dists  # = semantic^0.5 * visual^0.5

  epoch 1：ViT 视觉特征未被破坏，visual_probs 已经很好（74% 准确率），语义对齐 semantic_probs 也基于高质量 ViT 文本特征，融合结果 = 25.8%。

  epoch 2+：微调后 ViT 视觉特征被破坏，visual_probs 下降，融合结果 ≥ 30%。

  由于 text_features_test 是在 __init__ 中以 torch.no_grad() 计算并冻结的（未注册为 buffer），它不随训练变化，但视觉特征的退化使语义对齐路径也失效。

---
  三、与 RN50 的对比总结

  初始状态（epoch 1，零样本）：

  RN50：视觉特征弱，OTAM 仅 60% 准确率
         ↓ 训练时，有充足"改进空间"，梯度信号有效
         → val 误差从 35.5% 稳定下降到 26.7%

  ViT：视觉特征强，OTAM 已 74% 准确率
       ↓ 训练时，LR=1e-5 过大，全局注意力特征被扰乱
       → val 误差从 25.8% 升到 30%，慢慢恢复但永不破 25.8%
       → min_top1_err 被永久锁定在 epoch 1 的值

  min_top1_err 从不改变的直接机制：min_top1_err 追踪历史最低验证误差。对于 ViT，历史最低点永远是第 1 轮（训练还未破坏特征的时候），之后的每一轮 top1_err 都 ≥ 25.8%，所以 min_top1_err
  永久固定在 25.826%。

---
  四、可能的改进方向（仅供参考，不涉及代码修改）

  1. 降低 ViT 的学习率：例如 BASE_LR: 0.000001（1e-6，比 RN50 的 1e-5 小一个数量级），减少微调对 CLIP 特征的破坏
  2. 冻结 ViT backbone：只让 context2 和 query_embed_train 学习，backbone 保持 CLIP 预训练权重不变
  3. 使用更强的梯度裁剪：CLIP_GRAD_NORM 设一个较小的值（如 1.0）来限制 ViT 参数的更新幅度
  4. 使用更小的学习率热身：让 WARMUP 更长（如 WARMUP_EPOCHS: 3），让模型更平缓地从初始点开始训练
