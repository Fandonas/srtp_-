---
  发现的问题：val_meter 没有在每次 eval_epoch 前 reset

  关键代码路径（train_net_few_shot.py:78-82）：

  val_meter.set_model_ema_enabled(False)
  eval_epoch(val_loader, model, val_meter, ...)        # 第一次评估
  if model_ema is not None:
      val_meter.set_model_ema_enabled(True)
      eval_epoch(val_loader, model_ema.module, val_meter, ...)  # 第二次评估，meter 未 reset！

  val_meter.reset() 只在 eval_epoch 函数末尾（第454行）调用一次，但如果 model_ema 存在，会连续调用两次 eval_epoch，第二次调用时 meter 里还残留着第一次的统计数据。

---
  具体数值举例说明

  假设 5-way 1-shot，每次验证跑 10000 个 episode：

  第一次 eval_epoch（普通模型）：
  - num_top1_mis 累积了 10000 个 episode 的误差
  - num_samples = 10000
  - top1_err = num_top1_mis / 10000，假设结果 = 50%（准确率50%）

  第二次 eval_epoch（EMA模型），meter 未 reset：
  - num_top1_mis 在原来基础上继续累积 10000 个 episode
  - num_samples = 20000（！！）
  - top1_err = num_top1_mis_total / 20000

  如果 EMA 模型真实误差也是 50%，那结果仍是 50%，看起来没问题。

  但如果 EMA 模型更好（误差 40%），则：
  top1_err = (50% * 10000 + 40% * 10000) / 20000 = 45%
  报告的是混合平均值 45%，而不是 EMA 模型真实的 40%。

  反之，如果普通模型更好（误差 40%），EMA 模型误差 50%：
  top1_err = (40% * 10000 + 50% * 10000) / 20000 = 45%
  EMA 模型的结果被普通模型"拉低"了，报告 45% 而非真实的 50%，这就是"虚高"（准确率虚高）的来源。

---
  另一个潜在问题：mb_size 与实际 query 数不匹配

  meters.py:745：
  self.num_top1_mis += top1_err * mb_size   # top1_err 是百分比（0-100）
  self.num_samples += mb_size

  train_net_few_shot.py:417：
  val_loader.batch_size * max(misc.get_num_gpus(cfg), 1)

  少样本学习中 val_loader.batch_size = 1（每次一个 episode），但 top1_err 是基于 preds.size(0)（例如 75 个 query 样本）计算的百分比。

  mb_size = 1 时：
  - num_top1_mis = sum(各episode误差百分比)
  - num_samples = episode数量
  - 最终 = 各 episode 误差百分比的算术平均

  这本身是正确的，但如果 batch_size > 1（比如设为4），而代码里 task_dict[k][0] 只取了 batch 第一个 episode，mb_size
  却按4计算，权重被放大4倍但数据只有1份，最终结果仍是算术平均（因为分子分母同比例放大），不影响最终值。所以这个问题不是根因。

---
  结论

  根本原因：当 model_ema 存在时，第二次 eval_epoch 调用前 val_meter 没有 reset，导致两次评估的统计数据混合在一起，log_epoch_stats 输出的是两次评估的混合平均，而非单次模型的真实结果。

  验证方法：检查你的配置里 model_ema 是否启用。如果禁用 EMA，只跑一次 eval_epoch，结果应该是准确的。

---
  真正导致"结果虚高"的问题：QUERY_PER_CLASS_TEST: 1

  TRAIN:
    QUERY_PER_CLASS: 2      # 训练时每类2个query
    QUERY_PER_CLASS_TEST: 1 # 测试时每类只有1个query！

  5-way 1-shot，每类只有 1个 query，每个 episode 总共只有 5个 query 样本。

  top1_err 的计算：
  top1_err = (1 - correct / preds.size(0)) * 100
  # preds.size(0) = 5

  每个 episode 只有 5 个样本，correct 只能是 0、1、2、3、4、5 中的整数。

  所以 top1_err 只能取这些离散值：
  correct=0 → 100%
  correct=1 → 80%
  correct=2 → 60%
  correct=3 → 40%
  correct=4 → 20%
  correct=5 → 0%

  每个 episode 的误差粒度极粗（步长20%），统计噪声极大。10000个 episode 的平均值虽然会收敛，但单个 episode 的方差非常高，val_iter 的窗口中位数（50个episode的中位数）会在 40%/60%
  之间跳动，不能反映真实性能。

  对比标准设置：论文通常用 QUERY_PER_CLASS_TEST: 15（每类15个query，共75个样本），粒度为 1/75 ≈ 1.3%，统计更可靠。
