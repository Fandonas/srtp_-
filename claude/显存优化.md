  一、【最严重】self._full_clip_model 初始化后永久占用显存                                                                                                                                                                  
                                                                                                                                                                                                                            
  位置： semantic_alignment_few_shot.py → _ensure_text_features_initialized()

  二、【严重】forward 里 backbone 做了两次前向传播

  位置： semantic_alignment_few_shot.py → forward()

  问题：
  # 第1次：父类 forward 内部调用
  model_dict = super().forward(inputs)
  # → few_shot.py:2797 行：self.get_feats(support_images, target_images, ...)

  # 第2次：子类又调用一遍
  support_features, target_features, _ = self.get_feats(

  以当前配置为例（5-way 1-shot，4帧，QUERY_PER_CLASS=2）：
  - 每个 episode：(5+10) × 4 = 60帧
  - 两次前向传播 = 等效于 120帧 过 ViT-B/16
  - 训练时两套激活值都要保留用于反向传播 → 额外约 50~150MB 激活

  三、【中等】DATA_LOADER.NUM_WORKERS: 32 严重过高

  位置： 4frame_loss_p_vit.yaml:195

  DATA_LOADER:
    NUM_WORKERS: 32   # 32 个工作进程，每个都持有独立数据副本
    PIN_MEMORY: true  # 固定内存模式，放大 CPU 内存消耗

  对于 Few-shot 训练，每个 iteration 只处理 1 个 episode，32 个 worker 完全是资源浪费。建议降到 4~8，可显著减少 CPU 固定内存（pinned memory）占用。
